# Critical Loop 08：模型可用性与统一应用壳

## 评分纠正

当前可辩护分数：**78/100**。此前 92 分评分过高，原因是评审偏重功能清单和自动化测试，没有覆盖发布环境中的真实连接错误与跨页面视觉连续性。

## 本轮真实用户发现

- P0：模型请求使用 Edge Runtime 不支持的 `redirect: "error"`，导致配置正确也无法测试。
- P0：ETF、量化、交易复盘与个人工作台分别维护导航，入口顺序、品牌区和激活态可能不一致。
- P1：HKGAI 表单使用 Ollama 的 `qwen3:8b` 作为通用占位示例，诱导用户填写错误模型 ID。
- P1：零 Key 的确定性规则与需要推理服务的开源生成模型没有解释清楚。
- P1：保存默认 Provider 后会自动用于任务，但页面没有让用户确认这个行为。

## 已实施

- 改用 `redirect: "manual"` 并显式拒绝 3xx，兼容 Edge Runtime 且不跟随未知重定向。
- HKGAI 根地址自动规范为 `/v1`；通过 `/v1/models` 获取服务商实际模型 ID。
- 测试时若模型为空，先自动发现模型；连接成功后明确说明后续任务自动调用。
- 创建共享 `AppNavigation`，个人工作台与工具页使用同一导航数据、结构和激活态。
- 明确规则模式可零 Key；生成式开源模型仍需本机或服务器推理服务。

## 仍未完成

- 平台没有已部署的开源推理服务器，不能把规则模式称为生成式 AI。
- HKGAI 真实 Key 的线上连接与固定题集需要由有效凭证完成。
- 主股票研究页仍有内部状态导航，需要在视觉回归后决定是否完全迁移到共享壳。
- 尚无 10–20 位外部用户或真实付费行为证据。
