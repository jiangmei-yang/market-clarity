# Critical Loop 04

评审日期：2026-07-23

## 本轮评委质疑

1. 完成反馈表的人可以计数，但没有开始人数时，“完成率”从何而来？
2. 用户进入审查后立即点“稍后再看”，是完成任务，还是快速退出？
3. 付费候补人数没有曝光分母，如何判断价格假设？
4. 页面关闭、完成和网络请求乱序时，会不会把完成误记为放弃？

## 实际走查结果

- 以本地测试用户完成了“新建决策 → 查看数字风险 → 稍后再看 → 结果页”。
- 整段流程只用了 6 秒；它是合法的“延迟”选择，但不能被包装成深度完成。
- 证据中心如实显示 1 次开始、1 次完成、6 秒平均耗时和 0 份反馈，证明已有真实分母，也暴露了完成质量问题。
- 任务会话原实现只在 `started` 存在时更新完成；网络乱序会漏记。

## 已完成修改

- 新增匿名任务会话表，独立记录开始、完成、放弃和耗时，不保存投资输入。
- 会话终态改为幂等 UPSERT：完成可以在开始事件之前到达；迟到的放弃事件不能覆盖已完成。
- 证据中心同时显示完成率、放弃、平均耗时、“15 秒内快速结束”和“有参与度完成”。
- 15 秒阈值只用于识别点进即退，明确不代表投资判断质量。
- 价格实验新增登录用户唯一曝光、总访问、候补人数和转化率。
- 会话和完成后反馈可分别导出 CSV，避免将没有反馈的用户从分母中删掉。
