# Critical Loop 05

评审日期：2026-07-23

## 本轮对齐的课程要求

COMP 6211K 不只要求可点击的产品，还要求数据准备度、延迟、可靠性、缓存、失败模式和可复现的评测证据。本轮不增加新的投资功能，只验证数据底座。

## 评委质疑

1. 页面上某一只股票能显示，不代表数据服务稳定。
2. 你们说“有真实数据”，成功率、P95 延迟、来源覆盖率和缓存比例是多少？
3. 备用源、过期数据和新鲜缓存是否被准确区分？
4. 评委能否在产品里亲自重跑，还是只能相信团队准备的截图？

## 实现

- 新增 `/api/evaluation/data-sources`，按登录账户保存真实路由采样结果。
- `/evaluation` 新增可手动重跑的 10 只固定样本，覆盖沪市、深市、科创板、创业板和 ETF。
- 每只同时调用产品的行情路由与公告路由，不使用演示行情或模拟公告。
- 记录路由状态、延迟、数据时间、实际来源、资料数、缓存命中和来源覆盖。
- 服务端重新计算汇总，不保存客户端传入的“成功率”。
- 服务端锁定固定样本清单和顺序，并校验状态、延迟、时间与来源计数，不能通过替换样本或伪造汇总抬高结果。
- 修正统一 Failure Mode：新鲜备用源是 `degraded`，只有超过有效期才是 `stale`；两者均可继续保持 `allow_trade=false`。
- 公告缓存键升级为 v2，防止修复前的错误状态继续污染页面。
- 修复服务端 UTC 与浏览器 HKT 时间格式不一致导致的 React hydration 失败。

## 本地真实采样证据

采样时间：2026-07-23 05:20 HKT。本次数字只代表当时网络和本地环境，不代表长期 SLA。

| 指标 | 结果 |
| --- | ---: |
| 固定样本 | 10 只 |
| 产品路由请求 | 20 个 |
| 路由可用率 | 100% |
| P50 延迟 | 11 ms |
| P95 延迟 | 1,069 ms |
| 行情和公告均有来源 | 90% |
| 缓存命中 | 50% |
| healthy / degraded / stale / unavailable | 0 / 20 / 0 / 0 |

### 如何解读

- 100% 是“路由返回可用结果”，不是“主数据源 100% 健康”。
- 20 项全部是 degraded，因为当前主要依赖已标明的公开备用源或新鲜缓存。
- 90% 来源覆盖表明有 1 只样本没有同时获得行情和公告来源；这个缺口保留在证据页，不计为完全覆盖。
- 第一次采样曾因状态语义错误显示 50% 可用；失败导致了 Failure Mode 和缓存版本修复，不会从迭代记录中删除。
